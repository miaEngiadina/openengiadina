2 DONE Research: Storage and querying of linked data

CLOSED: [2019-07-24 Wed 18:11]

2.0.1 How can linked data be stored and efficiently queried?

In a Triplestore, a purpose-built database. Data is queried with the SPARQL query language.

2.0.2 What are the existing technologies? What will openEngiadina use?

There are plenty of triplestores available. openEngiadina will use Blazegraph as underlying database.

Blazegraph is a highly optimized triplestore (it can use a GPU to perform queries) that supports metadata about a triple itself via a mechanism called Reification Done Right (RDR). The ability to do so is very important for features like access control (see also initial experimentat to use RDR for access control).

The backend will be implemented in the Elixir programming language. The language is well suited for network servers, code can be reused from existing projects (Pleroma and CommonsPub) and libraries for working with Linked Data exist.

3 Research: Semantic queries on existing public ActivityPub data

By crawling existing ActivityPub servers for public data we may be able to:

  • Study the shape of data by analyzing links between posts and to outside data
  • Provide an interface to run semantic queries on the data (SPARQL)
  • Explore interesting queries and how new content could be discovered

Data could be either fed into a triple store with built-in SPARQL support or by performing federated queries Ã  la Linked Data Fragments.

An interface for perfoming queries might be very interesting and should be made public.

5 User interfaces for publishing linked social data

User interfaces are required that allow users to create content with appropriate metadata and links to existing data. This might be done by automatically infering appropriate links or providing an autocomplete function that searches for existing data. Inspiration may be taken from the autocomplete functions built-in to the Wikidata Query Service and Comunica Web Framework as well as from the dokieli editor.

6 Beyond touristic data: a decentrallized book reading network

To explore the ideas we could create a decentrallized social book reading network (similar to Goodreads) built on top of the to be developed backend, where users can tag, comment and discuss books. Books are an excellent example application for exploring user interface ideas and testing the backend, as the set of metadata and annotations is known (e.g. genre, ISBN, author, publisher) and exists for a huge corpus of books. Furthermore interesting semantic queries are very natural (e.g. "books with average rating over 4 stars that have been read or mentioned by people I follow in the last 3 months").

7.1 Clients & Applications

7.1.1 Data bridges

Import data from existing sources.

Potential data sources include:

  • Outdoor tours (e.g. from Gipfelbuch.ch, Hikr.org or outdooractive)
  • Weather data (e.g. from MeteoSwiss)
  • Touristic platforms (e.g. discover.swiss)

7.1.2 Distributed content management system

Various actors may publish content. From this published content, one specific actor, the curator, can select content that should appear on a web site.

Technically this may be implemented by defining a special collection of content that should appear on the web site and allowing the curator to add and remove content from this collection (with the ActivityStream Add and Remove activities). The web site is then generated from the content in this special collection.

Visitors of the web site do not need to know about the distributed mechanism in the background. They simply see the curated content.

7.1.3 Event calendar

An application that shows a calendar view of events in a specific region.

7.1.4 Map

An application that shows a map with points of interest. An initial version exists here.

7.1.5 Existing clients

There are existing ActivityPub clients (e.g. Fedilab and Whalebird). Research is required to what extent they can be used for openEngiadina (what content types do they support?).

7.2 Backend

7.2.1 ActivityPub Server to Server interactions

Enable federation with other ActivityPub services.

7.2.2 Content moderation

Given the distributed manner of content creation it is important that operators of backends have the ability to moderate content. Research into existing tools for moderation available in current ActivityPub implementaitons and what kind of tools are needed for openEngiadina is required.

7.2.3 Content discovery & searchability

The ability to discover and search knowledge is extremely important. Further research in how this can be done (and is done in existing implementations) is required.

Insight might be gained by studying existing frameworks for linked data and information retrieval such as Apache Jena, SPARQL and Xapian.
